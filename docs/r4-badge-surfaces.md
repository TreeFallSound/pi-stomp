# R4 — Badge Surfaces, Space Budget, and Degradation

**Status:** Read-only research. No code modified.
**Scope:** v2/v3 hardware (color 320×240 LCD). v1's monochrome LCD is out of scope for the
charter's badge work; it has no custom plugin panels.

LCD: 320×240, 16-bit RGB565, pushed over SPI at 50 MHz actual on both Pi 3 (v2) and
Pi 5 (v3) per `uilib/spi_timing.py:33-37` and `tests/test_spi_timing.py:5-6`.

---

## 1. Summary

Every badgeable surface on the 320×240 LCD has at least one viable placement for a small
(~10×10 px) glyph near the parameter it labels: custom panels already render an *encoder
label* text inside each knob/arc/band cell (e.g. `LEVEL`, `BASS`, `B4`, `Band 3/10`), and
a numeric badge ①②③ can replace or sit beside that text without enlarging any cell. The
footswitch strip at `y=204..239` already reserves a 12-px-diameter *letter-badge* circle at
the top of each slot (`uilib/footswitch.py:48-52`), so `(A)–(D)` is a near-zero-cost swap
of the existing `_draw_letter_badge` path.

The two worst-case dense panels — the 10-band graphic EQ
(`plugins/eq/graphic.py:28-38`) and the parametric EQ
(`plugins/eq/parametric.py:42-65`) — have **no spare horizontal pixels per band**
(32 px columns / 4-px nodes) but do have a 22-px *readout strip* at the top that already
displays the selected band's identity and value. The same strip can carry the ①②③ badge
for the *selected* band (the only band a tweak encoder acts on at any given moment), so
the dense graph stays untouched. Unselected bands never need a tweak badge because the
binding is selection-dependent and currently inactive for them.

Render cost is dominated by the fixed per-SPI-call overhead (0.71 ms,
`uilib/spi_timing.py:98`); a single 10×10 badge is **0.75 ms** at 50 MHz and a full
footswitch-slot repaint (36×36) is **1.20 ms** — both well under the 8 ms
`INLINE_BUDGET_MS` inline-push gate (`uilib/panel.py:390`) and the 10 ms tick.
Requirement 6 holds with margin.

## 3. Worst-Case Panel Analysis

### 3a. 10-band graphic EQ (`plugins/eq/graphic.py`)

Geometry (`graphic.py:28-38`):
```
COL_W = 320 // 10 = 32 px per column
BAR_W = 3 px
BAR_Y0 = 6, BAR_Y1 = 164  → bar area 158 px tall
FREQ_LABEL_Y = 172, FREQ_LABEL_H = 14 → freq labels 172..186
WIDGET_H = 186 (incl freq labels)
READOUT_H = 22 (above the bar widget)
```

Per-column budget: **32 px wide**. A 10-px badge centred on the column at `cx-5..cx+5`
would span 10 of 32 px, but it would also overprint the *3 neighbour bars* (each at
`bar_x = cx-1..cx+2`) and their `HALO_R=6` selection halos. A 10×10 badge cannot physically
fit on a single bar column without destroying the graph.

The selection halo itself is `NODE_R+2 = 6` px radius (12 px diameter) — **the halo *is* the
visual indicator today**. The badge ① for "tweak enc1 = selected band's gain" attaches to
the *readout strip*, which shows `Band 3/10  1.2k  +6.0 dB` and has roughly 150 px of empty
space to the right of those strings (snapshot shows the readout's right side is fully dark
past the gain text).

**Math:** readout strip is 320×22 = 7040 px. The current content per
`GraphicReadoutWidget._draw` (`graphic.py:248-262`) uses ~30 px (`"Band 3/10"`) on the
left, ~40 px (freq) centred, ~30 px (gain) at the right — ~100 px of content, leaving
~220 px across the strip's three gaps. A 10×10 badge at e.g. `x=0, y=0` (top-left of the
strip) is unambiguous and costs 10×10 = 100 px of dirty-rect.

**Conclusion: badges fit on this worst case by moving to the readout strip, not the bars.**
This matches the binding's selection-dependence: only the selected band is editable, so
only its badge is shown. Unselected bars never need a tweak badge.

### 3b. Dense parametric EQ (`plugins/eq/parametric.py`)

Geometry (`parametric.py:42-65`):
```
GRAPH_Y0 = 22, GRAPH_Y1 = 200  → 178 px tall graph, full 320 px wide
NODE_R = 4, HALO_R = 6  → band nodes are 8 px circles with 12 px selection halos
READOUT cols at x = 6 (name), 60 (freq), 160 (q), _W-6=314 (gain right-aligned)
```

Up to 6 bands on screen (a 6-band parametric). Nodes can sit anywhere along the
log-frequency x-axis (`freq_to_x`, `curve.py`), with no per-band column reservation.

Per-node budget: a 10×10 badge centred on a node *would* fit over the node (which is 12 px
diameter halo today) — but only for the *selected* band; the others aren't being tweaked.

The readout strip's 4 fixed columns have inter-column gutters: between `name` (x=6..~30)
and `freq` (x=60..~95) lies ~30 px (from x=30 to x=60); between `freq` (~95) and `q`
(x=160..~190) lies ~65 px; between `q` (~190) and `gain` (right-aligned at 314..~290) lies
~100 px. A 10×10 badge prepended to the `name` column at `x=6, y=6` rides the readout
refresh, exactly as in the graphic EQ.

**Conclusion: same as graphic EQ — badges live in the readout, only the selected band is
badged.** The graph stays legible.

## 4. Physical-Correspondence Rule

The charter says "badges should appear nearest the actual encoder/footswitch physical
position." Mapping on each hardware variant:

### v3 (L-shaped encoder block + 4 footswitches in a row)

Physical layout: all four encoders sit to the right of the LCD in an L. Top row, left to
right: `①(1) Tweak1, ②(2), ③(3)/Volume`. Bottom row: `NAV`, aligned beneath Tweak1 — so NAV
is *not* leftmost in a row with the tweaks; it's below and to the left of them. (The
on-screen reasoning below uses the user-confirmed physical layout; `pistomp/pistomptre.py`
only assigns encoder ids/pins, not position.) Footswitches below the LCD, left-to-right
`FS0, FS1, FS2, FS3` — the strip is "a linear LED strip mirroring the physical hardware"
(`lcd320x240.py:752-753`, slot order in `uilib/footswitch.py`).

**Today's screen mapping:**
- Knobs / arc dials in custom panels are drawn in left-to-right column order matching
  encoder-id order (e.g. gx_cabinet: `col0=LEVEL, col1=BASS, col2=TREBLE`,
  `gx_cabinet/panel.py:101-135`; the tweak dispatch in `on_event` uses `encoder_id == 2`
  before `3`, so enc2 → leftmost knob, enc3 → next, etc.). **So a ① glyph already reads
  "this knob is bound to tweak1"**, in the same left-to-right order as the physical
  encoders.
- The footswitch strip at `y=204..239` lays out slots in `fs.id` order, left to right
  (`lcd320x240.py:757-779`). The dot in slot `i` is at `x = pitch*i + slot_w/2`, so the
  on-screen footswitch row **physically corresponds** to the hardware row. The letter badge
  `(A)..(D)` placed on that dot is naturally nearest the physical switch it represents.

**Rule for badges on v3:**
- **Tweak encoder ①(②③):** place to the left of / above the knob or band label that the
  encoder binds to. If the panel renders knobs in left-to-right encoder-id order (the
  existing convention), the badge's x-order matches the physical encoder order.
- **Footswitch (A)..(D):** place on the footswitch slot's existing badge disc (slot `i →
  (A+i)`). The strip already preserves physical-left-to-right order.

### v2 (single NAV encoder + footswitches, no tweak encoders)

v2 has no tweak encoders, so ①②③ are N/A *on this hardware*. But the charter requires
v2 parity: NAV-only users edit through the selection model and `Parameterdialog`, so a
badge's *information* on v2 is "this parameter is bound to *some* physical encoder on v3;
on v2 you edit it via NAV." The NAV selection highlight **is** NAV's badge (charter A3:
NAV exclusion is an axiom, not a policy). So on v2:
- A parameter-list row that has a tweak binding on v3 still shows ① in the row's
  left margin — purely informational, telling the v2 user "this is bindable, edit via
  NAV." The `①②③` glyph clarifies which v3 encoder the binding corresponds to, which v2
  users ignore.
- Footswitch `(A)..(D)` badges are equally applicable on v2 — same physical row, same
  strip, same `FootswitchWidget._draw_letter_badge` code path.

**v2 physical correspondence:** footswitch physical order → screen slot order is the
same identity as v3. There is no "nearest encoder" question because v2 has only NAV
(which is unbadged).

## 5. Badge Geometry Spec

### 5.1 Tweak-encoder badge ①②③ (Unicode U+2460..U+2468)

- **Size:** 10×10 px (1 px padding inside a 12-px disc if rendered as a circled-number
  glyph; or bare 10×10 glyph if the existing 11-pt `arc_label` DejaVuSans-Bold font
  already fits U+2460 at ~10×10 — confirmed: `uilib/config.py:81-82` defines
  `arc_label=11pt Bold`, and `uilib/glyphs/arc_dial.py:209-216` uses it for knob
  labels).
- **Shape:** the circled-number Unicode glyphs in the existing `arc_label` font; the font
  already renders those codepoints as a clean circle around the digit. No new glyph asset
  needed.
- **Colour:** encoder ① → accent A, ② → accent B, ③ → accent C (reuse the panel's
  existing knob colour conventions — e.g. gx_cabinet: `COLOR_LEVEL`,
  `COLOR_BASS`, `COLOR_TREBLE`, `panel.py:28-30`). Tinted via `tint_mask` like the
  existing `CircleGlyph` paths.
- **Placement priority (per surface):**

  | Surface | Position |
  |---|---|
  | Compressor / gx_cabinet / tap_reverb / multiband | Prepend to the existing knob label text (no layout change). |
  | Graphic EQ / parametric EQ | Readout strip, top-left at `(0, 0)` in readout coords, only for the *selected* band. |
  | Parameter-list menu row | Row-local `(2, 2)` (left margin). |
  | `Parameterdialog` | Dialog-local `(2, 4)` (top-left of the body, below the title strip). |
  | Edit-in-place | Top-right of the selected value widget's bounding box. |
  | NAM setup view | Prepend to the knob label. |
  | NAM capture view / `notes` / tuner | **No badge** (no binding or modal-no-badge per A2). |

### 5.2 Footswitch badge (A)..(D)

- **Size:** 12×12 px (current `BADGE_DIAMETER = 2 * BADGE_RADIUS = 12`,
  `footswitch.py:49-50`). Letter glyph in `footswitch_badge` 14-pt Bold
  (`uilib/config.py:71-72`).
- **Shape:** a filled circle with the letter inside, exactly as
  `FootswitchWidget._draw_letter_badge` (`footswitch.py:207-220`) already renders.
- **Colour:** `BADGE_ON_FILL = (130, 130, 130)` when active, `BADGE_OFF_BORDER =
  (50, 50, 50)` when off (`footswitch.py:74-75`). When the footswitch is *bound* to a
  plugin parameter, replace the fill with the plugin's category colour (the existing
  `color` parameter to `FootswitchWidget`, `lcd320x240.py:770`).
- **Placement:** slot centre `cx = slot_w // 2`, `cy = BADGE_CENTER_Y = 10` — unchanged
  from current. The letters `A`–`D` replace the current "letter badge for unassigned
  switches" semantics; the binding-table renderer decides whether a slot shows a dot+label
  (existing bound behaviour) or an `(A)` glyph (unassigned, with no binding yet, plus
  shadowed/pierced contexts).

### 5.3 Generic MIDI badge (unknown external controller on the same CC)

Per charter A5: "Bindings we cannot name (external controllers on the same CC space) get
a generic MIDI badge, not a guess."

- **Shape:** 8×8 px circle (a small filled disc with no number, drawn via
  `CircleGlyph(4)` — `uilib/glyphs/circle.py`).
- **Colour:** dim grey `(90, 90, 110)` — distinct from the labelled ①②③ palette.
- **Placement:** same as the tweak badge for that row / knob, but only appears when the
  handler's binding table cannot resolve a known controller for a CC.

## 6. Degradation Ladder

The trigger conditions are, in priority order (first match wins):

| Level | Trigger | What renders |
|---|---|---|
| **L0 — Full badge** | The handler's effective binding table has a *named, currently-live* binding for the parameter on a known controller (tweak 1/2/3 or footswitch A/B/C/D). | The full ①②③ or `(A)..(D)` glyph in the placement of §5. |
| **L1 — Dot/Tick** | (a) The parameter has a binding but the controlling context is *shadowed* by a higher-priority context (charter R2 precedence); or (b) the surface's free space is too small for a labelled glyph (degradation threshold: a candidate position with `min(w,h) < 10`). Only graphic EQ bars and parametric EQ nodes qualify today; their degradation is to show the badge in the readout strip instead, not to a dot. The dot fallback is for the unlikely case where even the readout strip is full (e.g. a writable plugin with `Band 10/10` text + 3 gain digits + a long Q string). | A 4×4 px filled square in the same position as the full badge would have gone, tinted with the encoder colour. Conveys "something is bound here, identity not shown." Pairs with the readout strip text which, on next nav step, reveals which. |
| **L2 — Nothing + one-time coach mark** | The parameter has *no binding at all* in the effective table. Render nothing. The *first time* this happens on a given surface (per pedalboard-load), show a one-time coach mark: a 320×16 strip at the top of the panel saying `No encoder bound — NAV edits in place` for ~2 seconds. | Nothing inline. A short-lived full-width strip pushed as a decorator on the panel, with its own dismissal timer. |
| **L3 — Generic MIDI badge** | (Parallel to L0.) The parameter is bound CC-wise but the binding table cannot resolve a named controller (external MIDI on the same CC, per R3). | The generic grey disc of §5.3. |

**Coach-mark counter** lives in `settings.yml` under key `coachmark.badge.<surface-id>`,
an integer counting how many times the coach mark has been shown for that surface.
Incremented each time L2 fires; once it reaches `3` (configurable), the coach mark stops
showing for that surface in that pedalboard. The settings file path is
`/home/pistomp/data/config/settings.yml` (`pistomp/settings.py:21-22`), and the existing
`Settings.get_setting`/`set_setting` pair (`settings.py:46-60`) is the API.

Counter key scheme: `coachmark.badge.<panel-class>.<plugin-instance-id-or-"main">`.
Resets when the pedalboard changes (the value is keyed per-pedalboard via the
plugin-instance-id component, since pedals are reloaded on pedalboard change).

## 7. Render Budget Analysis

The inline-push gate (`uilib/panel.py:518`, `INLINE_BUDGET_MS = 8.0` ms,
`uilib/panel.py:390`) decides whether a dirty rect is pushed inline in the 10 ms tick or
coalesced into a deferred flush. Anything under 8 ms goes inline; over that coalesces into
the poll slot. Below are per-badge costs computed with `uilib/spi_timing.transfer_ms`
(`spi_timing.py:101-104`), using the fit constants from `spi_timing.py:91-98` and the
production 50 MHz actual clock (`test_spi_timing.py:91-95`).

### 7.1 Single-badge repaint

| Repaint | Pixels | Cost @ 50 MHz | vs 8 ms budget |
|---|---:|---:|---:|
| 10×10 tweak badge | 100 | **0.750 ms** | 9.4% |
| 12×12 footswitch badge | 144 | 0.766 ms | 9.6% |
| 20×16 readout-strip badge (EQ panels) | 320 | 0.833 ms | 10.4% |
| Selection ring (existing baseline) | 2560 | 1.681 ms | 21% |

The fixed per-call overhead `FIXED_MS = 0.71 ms` dominates (`spi_timing.py:98`); the
per-pixel pipeline cost is negligible at these sizes. So a single badge repaint ≈ 0.75 ms
regardless of whether it's 10×10 or 12×12.

### 7.2 Worst-case per-tick cost

Worst realistic per-tick scenario: NAV moves to a new row in the parameter menu, the
selection ring repaints (old + new = 2× ring ≈ 3.36 ms), and the new row's badge paints
(0.75 ms), and a co-occurring footswitch toggles (footswitch strip repaint = 1.2 ms):

```
selection ring (2×):    3.36 ms
new-row badge:          0.75 ms
footswitch slot repaint: 1.20 ms
────────────────────────────────
total:                  5.31 ms  (53% of 10 ms tick, 66% of 8 ms inline gate)
```

All three fit inline under `INLINE_BUDGET_MS=8.0`. The 10 ms tick has ~4.7 ms to spare for
the rest of the loop (`poll_controls`, `poll_ws_messages`, etc.). The LCD cadence itself is
every ~80 ms in steady state (`CLAUDE.md` loop table), so the badge repaint isn't on every
tick.

### 7.3 Worst-case panel-open cost

Opening a compressor panel with 4 arc badges prepended to labels: 4× badge paints happen
in a single full-panel refresh, which is a structural push that already flushes the
whole 320×240 (~30 ms, coalesced). The 4 badges add 4×100 = 400 px to a full-screen
push — invisible against the 76,800 px full-screen cost.

### 7.4 Edit-mode state changes (charter A4)

Edit-mode entry repaints the selection ring (already 80×32 = 2560 px, 1.68 ms) plus a
single badge (0.75 ms). The edit-mode state-change tail (value styling) is contained in
the ring's existing dirty rect. Total ≈ 1.7 ms + 0.75 ms = 2.45 ms, comfortably inline.

### 7.5 A worst-case stress: 10-band graphic EQ with a fast NAV spin

Moving selection across 10 bands in one tick: each selection step already repaints the
selection ring on the bar widget (≈ 320×8 px per row transition, ~0.9 ms each — the
`set_selected` path in `graphic.py:140-143`). Coalesced via `Panel._select_widget_ref`'s
`stack.batch()` (`uilib/panel.py:114-131`), the batch flushes once. Adding a readout-strip
badge repaint per step: it coalesces into the same batch, since the readout is repainted
on every `_select_widget_ref` (`graphic.py:471-485`). Net new cost per step: ~0.75 ms
from the badge's own dirty rect, folded into the batched flush — still under 8 ms.

**Requirement 6 verdict:** *Badge paints and edit-mode state changes stay within the
dirty-rect/SPI budget on the 10 ms tick.* **Holds.** The maximum single-tick inline cost
observed is ~5.3 ms (selection + badge + footswitch), leaving 47% headroom on the 10 ms
tick and 34% headroom under the 8 ms inline gate.

## 8. Recommendation

Adopt the following badge spec:

1. **Tweak encoder badge:** 10×10 circled-number glyph rendered in the existing `arc_label`
   (11 pt DejaVuSans-Bold) font, tinted with the panel's existing per-encoder accent
   colour. **Prepend to the existing knob label text** on compressor / gx_cabinet /
   tap_reverb / multiband / NAM-setup panels. **Render in the readout strip** for graphic
   and parametric EQ, attached to the *selected band only* (the binding is
   selection-dependent). **Left-margin** on parameter-list menu rows.
   **Top-left of dialog body** for `Parameterdialog`. **Top-right of the selection's
   bounding box** for edit-in-place.

2. **Footswitch badge:** reuse the existing `FootswitchWidget._draw_letter_badge` disc at
   `BADGE_CENTER_Y=10, BADGE_RADIUS=6`, swapping the current unlabeled placeholder for an
   `A`/`B`/`C`/`D` letter in `footswitch_badge` 14 pt Bold. Tint the disc fill with the
   bound plugin's category colour (existing `color` plumbing in `draw_footswitches`,
   `lcd320x240.py:770`).

3. **Generic MIDI badge:** 8×8 plain grey disc, same placement as the tweak badge it
   substitutes for. Rendered via `CircleGlyph(4)` and `tint_mask` exactly as the existing
   node-dot renderers in `parametric.py:68-76`.

4. **Physical-correspondence rule:** badges follow the panel's existing left-to-right
   encoder-id ordering (already the convention in `gx_cabinet`, `tap_reverb`,
   `compressor_base`'s tweak dispatch). Footswitch badges always render at the slot that
   matches the physical switch id, preserving the strip's "mirror of the physical row"
   invariant. On v2 the ①②③ badges are informational only (text labels "bindable via NAV").

5. **Degradation ladder:** L0 full badge (named live binding) → L1 4×4 dot (shadowed or
   <10 px slot) → L2 no badge + one-time top-strip coach mark (no binding, counter
   `coachmark.badge.<panel>.<plugin>` in `settings.yml`, capped at 3 shows per
   panel/pedalboard). L3 generic-MIDI badge in parallel with L0 for unnamed CCs.

6. **Render budget:** verified — single-badge repaint 0.75 ms, worst-case combined-tick
   repaint 5.31 ms, both under the 8 ms `INLINE_BUDGET_MS` gate and the 10 ms tick. No
   architectural change to the dirty-rect pipeline required.

7. **Implementation hooks:** the badge renderer reads *only* from the handler's effective
   binding table (charter A5 / requirement 5). It hooks into:
   - `Panel.do_draw` / widget `_draw` paths where labels are already rendered (prepend to
     label string for knob-based panels — minimal diff);
   - `GraphicReadoutWidget.set_fields` / `ReadoutWidget.set_fields` for the EQ panels
     (extend `set_fields` to accept a `badge` argument);
   - `FootswitchWidget.__init__` to receive the badge letter from the binding table
     instead of the current "unassigned" heuristic;
   - `Menu` row rendering to prepend a 10×10 glyph to the row label when the parameter
     is bound;
   - `Parameterdialog._draw_contents` for the single top-left badge;
   - The selection-ring repaint in `Panel._select_widget_ref` (the existing
     `stack.batch()` already coalesces the two rects — extend the batched region to
     include the top-right badge slot for edit-in-place surfaces).

No new widget class is required; the badges are paint-time additions to existing widgets,
driven by the binding table owned by the handler.
