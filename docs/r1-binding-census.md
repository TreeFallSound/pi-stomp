# R1 — Binding census

The complete inventory of input → effect pathways across pi‑Stomp v2 and
v3, each classified as static pedalboard-scoped, static panel-scoped,
selection-dependent, or modal/imperative, and tagged *declarable* or
*escape hatch*. This table is the migration checklist for requirement 4
of the charter.

All line references are to the `refactor/input-sink-transport` branch
that the charter starts from. Every pathway below was traced through
`Handler.handle` → `Lcd.handle` → `PanelStack.current` →
`Panel.handle` → `Panel.on_event` and the handler’s own
`_handle_*` branches; theLCD/blend/handler cascade order is the
canonical shadowing order (see §5).

---

## 1. Summary

pi‑Stomp v2/v3 routes every hardware input through one `InputSink`
cascade: the LCD asks its top `Panel` first (which dispatches NAV to
the base selection machinery and only offers non‑NAV events to
`on_event`), then an active blend mode gets a chance, then the
`Modhandler` runs event-type logic (parameter commit + MIDI emit, audio
card, footswitch/chord dispatch). Today’s input→effect pathways fall
into four clean buckets: (a) pedalboard-scoped static bindings formed by
TTL MIDI-learn + config overlay, (b) static panel-scoped bindings where
an open plugin panel claims a specific tweak encoder id for a named
symbol, (c) selection-dependent bindings where a tweak encoder edits
whatever widget NAV has highlighted, and (d) one genuine state machine
(the NAM capture flow). Footswitches already pierce the panel layer by
construction, NAV is already constitutionally unhijackable on this
branch (A1 is applied), and blend mode claims whole controllers by
identity rather than by event type. Twenty of the twenty-two pathways
are *declarable*; two (NAM capture, the multiband-menu slot identity
swap) carry just enough state to needn’t — see §6.

## 4. v2 vs v3 analysis

What exists on v2 today:

- **Navbar machinery (#1–3, #37–40):** the single NAV encoder + NAV click + NAV long‑press. All menu/panel navigation, all menu actions, and the main‑panel NAV grid operate identically on v2. The plugin‑bypass toggle (#38) is reachable on v2.
- **Analog MIDI binding (#27, #28):** potentiometer / expression pedal → CC, with TTL MIDI‑learn overlay — the *only* continuously variable plugin‑parameter input on v2.
- **Footswitches (#30–35):** all four footswitch paths (short press CC, preset change, tap tempo, relay bypass, long‑press chord) exist on v2; v2 has 3 footswitches, a physical relay, and the default config wires relay + long‑press chords.
- **Blend mode on a pot (#41):** the primary blend input on v2 is the expression pedal / main pot (an `AnalogMidiControl`); since plugin panels never consume `AnalogEvent`, blend works on v2 without being shadowed.

What is v3‑only with **no v2 equivalent today**:

- Every tweak encoder pathway (#4–#26) — by construction, since v2 has no tweak encoders (`pistomp/pistompcore.py:85` raises `NotImplementedError` for `add_encoder`). The A4 plan (edit‑in‑place on the base `Panel`) is what *creates* the v2 equivalent: each panel will gain a NAV‑driven edit‑in‑place state so that #5/#6/#8/#9/#11/#12 etc. become reachable through NAV alone.
- After A4 lands, the v2‑equivalent of #4, #7, #10, #13, #16 (selection‑dependent enc 1) is exactly “NAV‑click on the selected widget enters edit; NAV rotation edits the value through the same step grid.” Likewise the static panel‑scoped enc 2/3 bindings (#5, #6, #8, #9, #11–#15, #20–#21) only become NAV‑reachable **when the selection has landed on the matching widget** — i.e. #5 becomes a *selection‑dependent* binding on v2 (you must be on the `thr` arc to edit `thr`), even though on v3 it remains static (enc 2 always edits `thr` regardless of selection). **This is a real v2/v3 asymmetry R2 should acknowledge:** the same plugin panel presents *more* v3 bindings (static + selection‑dependent) than v2 bindings (selection‑dependent only).
- NAM capture flow (#17–#24): no v2 equivalent for the enc 2/3 audio‑nudge paths. **The NAM panel itself is fully NAV‑operable on v2** — enc 1 is just an alias for NAV (and the panel only depends on NAV for setup‑view navigation and the `Start` button). The audio‑nudge accelerators (#20–#21) are conveniences; their v2 fallback is “adjust input gain / output volume through the System menu”. Not a blocker.

Net statement: **every pathway that needs a v2 fallback is a tweak‑encoder pathway, and every one of those has a clean A4‑shaped NAV‑equivalent through edit‑in‑place**, except the “accelerator” alias routes (#17–#19) which are v3‑only conveniences and (#20–#21) which have a system‑menu equivalent on v2 today. No pathway requires a new control class on v2.

## 5. Shadowing behavior today

The cascade in `Modhandler.handle` (`modalapi/modhandler.py:248-263`) defines a single, total order:

1. **LCD / PanelStack.top** (`pistomp/lcd320x240.py:270`): only `pstack.current` is asked — the panel stack is *not* walked top‑down (the `README` describes a top‑down walk but the implementation only consults `current`). The current panel’s `Panel.handle` then:
   1. NAV `EncoderEvent` / NAV `SwitchEvent` are dispatched directly to the base selection machinery (`uilib/panel.py:211-219`). **`on_event` never sees a NAV event** (the A1 invariant — enforced, not conventional).
   2. `Panel.on_event(event)` is called for everything else (`uilib/panel.py:220`). The base implementation returns `False`; plugin panels override and consume any `EncoderEvent` whose `controller.id in (1, 2, 3)`. **No panel consumes `AnalogEvent` or footswitch `SwitchEvent`.**
   3. If neither matched, a KNOB/VOLUME `SwitchEvent` PRESS is mapped to `input_event(CLICK)` as a convenience (`uilib/panel.py:222-228`) — this is the “tweak knob click == NAV click” alias (#26).
2. **Active blend mode** (`modalapi/modhandler.py:254`, `blend/manager.py:181`): `intercept` claims an event iff `event.controller is self.input_controller` (identity, not event type) and the event is `AnalogEvent` or `EncoderEvent` (`blend/input_controller.py:95-112`). Blend mode does not see footswitches; it can be shadowed by an open panel whose `on_event` matches its input encoder.
3. **Handler own dispatch** (`modalapi/modhandler.py:256-262`): by event type — `_handle_encoder`, `_handle_analog`, `_handle_switch`.

**Per-control-class conflict resolution:**

- **NAV:** always consumed at the panel base layer; never seen elsewhere. Effectively a *policy* enforced by `Panel.handle` (post‑A1).
- **Tweak encoders (v3):** the **open panel wins**, then blend, then handler. There is no opt‑in/opt‑out flag — any panel that wants a tweak encoder just matches its id in `on_event`. The only dual‑source case today is enc 1 on a compressor‑equipped pedalboard where enc 1 also has a TTL MIDI‑learned CC binding (#36): panel open → panel wins; panel closed → handler commits to mod‑host.
- **Volume encoder (v3 enc 3, `type=VOLUME`):** routed by the handler as a special case (`Modhandler._handle_encoder:271-276`). **Subtlety discovered:** on a compressor panel open, enc 3 events have `controller.id == 3` and the compressor’s `on_event` claims id 3 (`plugins/compressor_base.py:106-107`) — so **the compressor panel shadows the volume encoder while it is open.** The same is true on the gx_cabinet panel (enc 3 → `CLevel`) and tap_reverb (enc 3 → `decay`). This means a v3 user with the default config (enc 3 = VOLUME) currently loses output‑volume control whenever one of these panels is open. Today the user simply doesn’t keep these panels open during play, but R2 should make this conflict explicit and either (a) let panels declare enc 3 overrides deliberately (and badge them) or (b) protect VOLUME from panel‑level shadowing by type, the way NAV is.
- **Pots / expression (v2 + v3):** `Panel.on_event` never matches them, so they **pierce the panel layer** by structural accident, not by an explicit rule. A3 makes “pots pierce panels” a *policy*: the policy is currently enforced by the *absence* of any `AnalogEvent` matcher in panels. R2 should write this as a positive rule (e.g. cascade order: analog → handler, skipping panel), not a negative one.
- **Footswitches:** `Panel.handle` does not match footswitch `SwitchEvent`s (NAV check fails, KNOB/VOLUME‑CLICK check fails because footswitch type is not KNOB or VOLUME), so they pierce the panel layer → handler `_handle_footswitch`. **Footswitches already pierce panels by construction today** — but, like pots, by structural accident rather than by an explicit opt‑in/opt‑out. A3 says panel‑level shadowing of footswitches is opt‑in; today no panel can opt in (no `on_event` matches a footswitch `SwitchEvent`). Enforcing the A3 rule means **adding the opt‑in path** without weakening anything.
- **Blend vs. handler vs. panel (encoders):** blend sits *between* panel and handler. A pedalboard that configures blend on enc 1 AND has a compressor panel open will lose blend control while that panel is open — the panel eats enc 1. Documented in §3 #41.
- **TTL MIDI‑learn vs. config overlay:** `Hardware.reinit` re‑applies the config overlay on top of the default and then `ControllerManager.bind` overlays TTL bindings on top of the controller table (`pistomp/controller_manager.py:67-101`). A CC configured in both `hardware.footswitches[*].midi_CC` and discovered via TTL MIDI‑learn will share one `controllers["{channel}:{CC}"]` entry — *last write wins*, and the TTL path (`_apply_midi_binding` at `pistomp/handler.py:224`) re‑binds at runtime when MOD‑UI broadcasts `midi_map`. This is the dual‑source truth question R3 owns; for R1 it is enough to note that **config and TTL are not unioned — they share the same slot and the order is “TTL runtime binding overwrites config binding.”**

## 6. Classification notes (declarable vs. escape hatch)

Most rows are plainly declarable: a `(control, event_kind, scope) → (effect target)` triple, where the effect target is a parameter symbol, an audio‑card symbol, a callback name, or a panel‑selection dispatch. The six that needed a second look:

- **#17–#19 (NAM enc 1 mirrors NAV):** *declarable*, as a single‑key alias in the schema — `(tweak1 rotation|click|longclick in NAM panel) → (NAV rotation|click|longclick)`. There is no state machine here; the alias is unconditional while the NAM panel is mounted. Conforms to A3: tweak encoders are bindable by contexts, and a context can say “this control behaves exactly like NAV.”
- **#22, #24 (NAM enc 2/3 swallowed during CAPTURING / DONE / ABORTED):** *this is the borderline case.* A pure declaration can carry a state predicate (`when state == CAPTURING`). If the R2 schema supports state‑conditional consumes, these are declarable; if it does not, they collapse back into the NAM escape hatch. **Recommendation for R2:** support an `enabled_when:` predicate on declarations so that the only genuine escape hatch in the system is the NAM capture state machine itself, not its event consumption.
- **#23 (NAM enc 2/3 fall‑through on FAILED):** declarable — the absence of a binding in this state *is* the declaration. Falls back to the standard parameter dialog overlay (`Modhandler._handle_encoder`).
- **#16 (multiband menu enc 1 → selected `ParamSlotWidget.on_encoder_rotation`):** selection‑dependent; the only thing distinguishing it from `#4` (compressor enc 1) is that the dispatch goes through `ParamSlotWidget.on_encoder_rotation` instead of `panel._edit_symbol`. Both are “edit the selected widget’s parameter” — declarable as the same schema key (`selection_dependent: true`).
- **#25 (tweak encoder long‑press → named callback):** declarable, but crosses a control class boundary (encoder button long‑press → footswitch‑style chord callback). Worth noting: today’s encoder long‑press does **not** enter the chord resolver (`FootswitchChords.observe` is only called from `_handle_footswitch` for `Footswitch` controllers at `pistomp/handler.py:140-142`), so the chord window semantics do not apply to encoders. The encoder long‑press fires its singleton callback immediately on the long‑press edge (`pistomp/handler.py:296-299`). R2 should clarify whether encoders ever want chord groups, or whether “encoder long‑press = singleton callback” is a policy.
- **NAM capture flow as a whole (#17–#24 plus the Start/Abort buttons):** the panel is the canonical *escape hatch* of the codebase. The state transitions (`IDLE → CAPTURING → DONE|FAILED|ABORTED → IDLE`) and the side effects on `routing.connect_monitor` / `_engine.start` / `_engine.abort` are not expressible as a binding set. **R2 must keep `on_event` available for this panel**, even after every *declarable* row has been migrated. The Start / Abort / Back / Retry buttons are standard `Button` widgets with `action=` lambdas — those are already standard selection‑CLICK actions, not a separate binding category, so they don’t need a row.

No row in the table is left “unknown.”

## 7. Open questions for R2

1. **State‑conditional consumes.** The NAM panel wants `(enc 2 in CAPTURING) → consume‑no‑op`. Without state predicates, this is an escape hatch; with them, it is declarable and the NAM panel *might* collapse entirely to declarations except for the state‑machine buttons. Decide before designing the schema.
2. **Cross‑control aliases.** NAM enc 1 → NAV (three aliases). The schema needs a way to say “this control behaves as if it were NAV,” cleanly, without leaking NAV through the binding table (A3 says NAV never gets a badge — the alias must not produce one either).
3. **Selection‑dependent declarations and badge rendering.** Rows #4, #7, #10, #13, #14, #15, #16 are selection‑dependent: the bound symbol changes as NAV moves. R4’s badge spec must show the *currently effective* symbol next to the encoder, refreshed on selection change. Does the declaration carry a “target = selected widget’s symbol” clause, or is the declaration per‑widget (one declaration per `BandSelectable`/`ArcSelectable`/`ParamSlotWidget`)? Both are viable; one of them is meaningfully more data‑like than the other.
4. **VOLUME encoder protection.** Should `type=VOLUME` be protected from panel‑level shadowing (like NAV), or should panels be allowed to override it with explicit on‑screen state (matching A3’s footswitch opt‑in rule)? The current behaviour (#36 caveat in §3, and §5) silently lets the compressor panel shadow the volume encoder. R2 should pick a rule and write the test that pins it.
5. **Consumes that do nothing (#11, #12, #22, #24).** A `consume: true, effect: none` declaration is meaningful — it prevents the event reaching the handler (and mod‑host) while a panel is open. Confirm R2 allows this and badge it consistently (a badge that fires nothing is the “honest” case the charter calls out in requirement 5).
6. **Encoder long‑press chord semantics (#25).** Should encoder long‑press ever enter the chord resolver, or is “encoder long‑press = singleton callback” a held policy? Today the two control classes diverge silently.
7. **Footswitch opt‑in panel shadowing (A3).** Today there is no opt‑in mechanism; the A3 rule must *add* one. R2 should specify exactly how a panel declares “I want footswitch N to be consumed by me while I’m open,” and the cascade point that checks it. The natural place is `Modhandler._handle_switch` consulting the effective binding table before falling back to `_handle_footswitch`.
8. **TTL + config + context merge order (R3‑shared).** Three sources (pedalboard TTL MIDI‑learn, config overlay, future context declarations) all want to write the same `(channel:CC) → effect` slot. R3 owns the merge rules; R2 only needs to know whether a context declaration can *shadow* a TTL binding or only *augment* an unbound one. The dual‑source case (#36) is live today and must not regress.
9. **Analog `autosync` on pedalboard load** (`pistomp/hardware.py:105-112`). This is a *binding‑time* side effect, not an input pathway, but the schema for an analog binding probably needs to carry an `autosync: true|false` flag so the new binding system can perform the same first‑load position sync without special‑casing `AnalogMidiControl`.
10. **`PANEL.handle` does not walk the panel stack** (`pistomp/lcd320x240.py:273`). Only `pstack.current` is consulted. If R2 wants a *context stack* (per the charter’s “context stack on top of the transport”), it must construct that stack explicitly; the current `PanelStack` is a paint stack, not an input stack. Not a blocker, but worth stating so R4 doesn’t assume a stack‑walk exists.
